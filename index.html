Pseudo-node 是 OSPF LSA2 對 Transit LAN 的抽象表示,實務上是一個虛擬節點。隱藏後只看實體 Router 拓樸。

故障操作

右鍵 點擊鏈路或 Router 節點 → 切換故障狀態。單擊節點可拖曳排版(Pseudo-node 為 LSA2 抽象,不可故障)

C1 路徑分析

計算 Source → Destination 最短路徑(SPT + ECMP),反映目前右鍵故障狀態,自動判斷 PRIMARY / BACKUP MODE。下方「無保護段掃描」檢查 primary 路徑上有無單點故障邊(SPOF)。
選擇 Source / Destination 後點擊「計算最短路徑」開始分析。

C2 最短路徑矩陣

全部 Router pair 的最短路徑 cost 一覽。底色深淺 = cost 大小; 右上 ²/³ = ECMP 路徑數;右下 * = 非對稱(與鏡像格 cost 不同)。 點任一格查看詳細路徑並在拓樸圖上高亮。

C3 樞紐度盤點

計算每條鏈路與每台節點的樞紐度(Betweenness Centrality):被多少 SPT 路徑經過,ECMP 等分計算。 樞紐度高 = 過路流量重,失效衝擊大;樞紐度 0 = 純備援,平時零過路流量。
分 4 階對應採購決策:樞紐擴容雙保 / 正常維持 / 偶爾降規候選 / 純備援裁併候選。

C9 邊流量視圖

每條鏈路目前實際扛多少 Gbps。
做法:把 demand.js 裡每對 Router 的流量需求放到最短路徑上(有 ECMP 時平均分攤),沿途每條邊累加。
與 C3 的差別:C3 數「有幾條路徑經過」(看結構),本頁算「實際帶幾 Gbps」(看流量) — 一條邊可能 C3 高但這裡低,表示常被用但每筆量小。
門檻(以最大邊負載為基準):> 60% 高 / > 20% 中 / 其餘低。

鏈路成本編輯

直接改 cost → 即時更新圖上標籤 + 自動重算路徑。非對稱 p2p 可分別設 →/← 兩方向。

C4 節點失效模擬

選一台 Router 模擬失效,重算全網 SPT。顯示連通性、分群,以及流量重分配明細(斷線 / ↑ 增加 / ↓ 減少)。本 Tab 自帶情境,獨立於右鍵故障狀態。

C5 ECMP 備援完整性檢查

⚠ 設計審計:基於完整拓樸,不受右鍵故障影響
若某 src → dst 有 多條等成本路徑(ECMP),理想狀況是「砍掉任一條 ECMP 鏈路,流量仍由剩下的 ECMP 鏈路接手」 — 不應該繞到 ECMP 群組外。本檢查掃描每對 Router,模擬 ECMP 邊一條條失效,看是否仍由群組內接手。
通過群組內任一邊失效後,剩餘 ECMP 邊仍能接手
失敗失效後不可達 或 流量繞到 ECMP 群組外
不適用本來就沒 ECMP(只有一條最短路、或多條共用同一出口)

C7 Prefix 宣告冗餘

⚠ 設計審計:基於完整拓樸,不受右鍵故障影響
基於 Prefix Index(§3.2 Rule 5):被 ≥2 節點宣告 = redundant(有備援),僅被 1 節點宣告 = single-advertised(單一宣告)。
每台 router 統計 single-advertised ÷ total 比例,依此著色:
= 0% — 全 prefix 都有冗餘宣告(理想) 0–33% — 少數 prefix 單一宣告 33–66% — 約一半 prefix 無冗餘 > 66% — 多數 prefix 單一宣告(高風險)

C8 N-1 Worst-case 排行

⚠ 設計審計:基於完整拓樸,不受右鍵故障影響
枚舉所有單點失效(每條 link、每台 router),對全網每對 pair 計算最差延遲。 排出全網最脆弱的 pair 與最致命的失效情境。 ∞= 該失效下變不可達。 ×N= 延遲變 N 倍。 點任一列在圖上可視化失效情境與重繞路徑。
